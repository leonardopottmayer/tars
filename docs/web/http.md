# Web.Http - Packages, DI and configuration

## Packages

### `Pottmayer.Tars.Web.Http.Abstractions`

Pure contracts, with no dependency on ASP.NET Core.

| Type | Role |
|---|---|
| `IHttpResponse` | base envelope with `Success` |
| `IHttpResponse<T>` | success envelope with `Data` |
| `IHttpErrorResponse` | error envelope with `ErrorCode`, `ErrorMessage` and `FieldErrors` |
| `IHttpFieldError` | individual field error |
| `IHttpErrorMapper` | maps `Error`/`Exception` to an HTTP response |
| `IPaginationInfo` | metadata for pagination headers |
| `IWrapDecisionService` | decides whether a response should be wrapped |
| `WrapDecisionContext` | context for the wrapping decision |
| `ControllersWrappingMode` | `WrapAll` or `WrapNone` |

### `Pottmayer.Tars.Web.Http`

Runtime implementations, still with no dependency on ASP.NET Core.

| Type | Role |
|---|---|
| `HttpResponse<T>` | success envelope with `Success`, `Data` and optional `TraceId` |
| `HttpErrorResponse` | error envelope with `Success`, `ErrorCode`, `ErrorMessage`, `FieldErrors` and optional `TraceId` |
| `HttpFieldError` | concrete implementation of `IHttpFieldError` |
| `DefaultHttpErrorMapper` | default mapper based on `ErrorType` and `IMessageProvider` |
| `WrapDecisionService` | default rule for when to wrap |
| `WebHttpOptions` | core options (`Enabled`, `IncludeTraceId`) |

### `Pottmayer.Tars.Web.Http.AspNetCore`

HTTP-host-specific integration.

| Type | Role |
|---|---|
| `TarsExceptionFilter` | MVC filter for expected and unexpected exceptions |
| `ResponseWrapperResultFilter` | MVC filter that wraps `ObjectResult`, `EmptyResult` and `NoContentResult` |
| `ResponseWrapperEndpointFilter` | endpoint filter for Minimal APIs |
| `ResponseWrapperMvcOptionsSetup` | registers the MVC filter globally via `IConfigureOptions<MvcOptions>` |
| `[EnableResponseWrapper]` | opt-in on controllers/actions when the default mode is `WrapNone` |
| `[DisableResponseWrapper]` | opt-out on controllers/actions when the default mode is `WrapAll` |
| `ResultHttpExtensions.ToHttpResult()` | converts `Result` and `Result<T>` to `IResult` |
| `ResponseWrapperEndpointExtensions` | `.AddTarsResponseWrapper()` and `.DisableTarsResponseWrapper()` |
| `PaginationExtensions.WritePaginationHeaders()` | writes `X-Pagination-*` to the response |
| `WebHttpAspNetCoreOptions` | host options (`ControllersDefaultMode`, `MinimalApisEnabledByDefault`) |

## Available DI

### Core

Minimal registration:

```csharp
builder.Services.AddTarsLocalization();
builder.AddTarsWebHttpOptions();
builder.Services.AddTarsDefaultHttpErrorMapper();
builder.Services.AddTarsDefaultWrapDecisionService();
```

Methods:

| Method | Role |
|---|---|
| `AddTarsWebHttpOptions(...)` | binds and validates `WebHttpOptions` |
| `AddTarsHttpErrorMapper<TMapper>()` | registers a custom mapper |
| `AddTarsDefaultHttpErrorMapper()` | registers `DefaultHttpErrorMapper` |
| `AddTarsWrapDecisionService<TService>()` | registers a custom wrapping rule |
| `AddTarsDefaultWrapDecisionService()` | registers `WrapDecisionService` |

Important note:

- `AddTarsDefaultHttpErrorMapper()` registers the default Tars messages in memory, but it still depends on `AddTarsLocalization()` for an `IMessageProvider` to exist.

### ASP.NET Core

```csharp
builder.AddTarsWebHttpAspNetCoreOptions();
builder.Services.AddTarsResponseWrapperResultFilter();
builder.Services.AddTarsResponseWrapperMvcOptionsSetup();
builder.Services.AddTarsResponseWrapperEndpointFilter();
builder.Services.AddTarsExceptionFilter();
builder.Services.AddTarsProblemDetails();
```

Methods:

| Method | Role |
|---|---|
| `AddTarsWebHttpAspNetCoreOptions(...)` | binds and validates `WebHttpAspNetCoreOptions` |
| `AddTarsResponseWrapperResultFilter()` | registers the MVC wrapping filter |
| `AddTarsResponseWrapperMvcOptionsSetup()` | adds the global MVC filter via options |
| `AddTarsResponseWrapperEndpointFilter()` | registers the filter for Minimal APIs |
| `AddTarsExceptionFilter()` | registers `TarsExceptionFilter` in DI |
| `AddTarsProblemDetails()` | customizes `ProblemDetails` with `traceId` |

To use the global exception filter:

```csharp
builder.Services.AddTarsExceptionFilter();
builder.Services.AddControllers(options =>
{
    options.Filters.AddService<TarsExceptionFilter>();
});
```

## appsettings

Both `Web.Http` options use the same section: `Tars:Web:Http`.

```json
{
  "Tars": {
    "Web": {
      "Http": {
        "Enabled": true,
        "IncludeTraceId": false,
        "ControllersDefaultMode": "WrapAll",
        "MinimalApisEnabledByDefault": false
      }
    }
  }
}
```

| Key | Type | Default | Description |
|---|---|---|---|
| `Enabled` | `bool` | `true` | turns wrapping on/off |
| `IncludeTraceId` | `bool` | `false` | includes `traceId` in the envelopes generated by the filters |
| `ControllersDefaultMode` | `string` | `"WrapAll"` | `WrapAll` wraps controllers by default; `WrapNone` requires `[EnableResponseWrapper]` |
| `MinimalApisEnabledByDefault` | `bool` | `false` | endpoints without explicit metadata can be wrapped by the endpoint filter |

Notes:

- the options methods accept `configure`, but the current classes use `init` properties; therefore, the practical path is `appsettings`/section binding
- `IncludeTraceId` does not affect `ToHttpResult()`, because that method builds the `IResult` directly

## Registration scenarios

### 1. Minimal for `ToHttpResult()`

```csharp
builder.Services.AddTarsLocalization();
builder.AddTarsWebHttpOptions();
builder.Services.AddTarsDefaultHttpErrorMapper();
```

Good for Minimal APIs where each endpoint calls `result.ToHttpResult(mapper)`.

### 2. Controllers with automatic wrapping

```csharp
builder.Services.AddTarsLocalization();
builder.AddTarsWebHttpOptions();
builder.AddTarsWebHttpAspNetCoreOptions();
builder.Services.AddTarsDefaultHttpErrorMapper();
builder.Services.AddTarsDefaultWrapDecisionService();
builder.Services.AddTarsResponseWrapperResultFilter();
builder.Services.AddTarsResponseWrapperMvcOptionsSetup();
builder.Services.AddTarsExceptionFilter();

builder.Services.AddControllers(options =>
{
    options.Filters.AddService<TarsExceptionFilter>();
});
```

### 3. Minimal APIs with per-group wrapping

```csharp
builder.Services.AddTarsLocalization();
builder.AddTarsWebHttpOptions();
builder.AddTarsWebHttpAspNetCoreOptions();
builder.Services.AddTarsDefaultWrapDecisionService();
builder.Services.AddTarsResponseWrapperEndpointFilter();

var api = app.MapGroup("/api").AddTarsResponseWrapper();
api.MapGet("/orders", () => new { items = Array.Empty<object>() });
api.MapGet("/health", () => new { ok = true })
   .DisableTarsResponseWrapper();
```

### 4. Minimal APIs with wrapping on by default

```json
{
  "Tars": {
    "Web": {
      "Http": {
        "MinimalApisEnabledByDefault": true
      }
    }
  }
}
```

Even with this flag, responses that are already `IResult` remain without additional wrapping.

### 5. `ProblemDetails` as the format for some endpoints

```csharp
builder.Services.AddTarsProblemDetails();

app.MapGet("/problem", () =>
{
    return Results.Problem(
        detail: "A downstream dependency failed.",
        statusCode: 503);
});
```

This scenario works well when the endpoint returns `IResult` directly. In MVC, `ProblemDetails` can still be wrapped by the `ResponseWrapperResultFilter`.

### 6. Custom mapper

```csharp
public sealed class AppHttpErrorMapper : IHttpErrorMapper
{
    private readonly IMessageProvider _messages;

    public AppHttpErrorMapper(IMessageProvider messages)
        => _messages = messages;

    public int MapToStatusCode(ErrorType errorType) => errorType switch
    {
        ErrorType.NotFound     => 404,
        ErrorType.Validation   => 400,
        ErrorType.Business     => 422,
        ErrorType.Unauthorized => 401,
        ErrorType.Forbidden    => 403,
        ErrorType.Conflict     => 409,
        _                      => 500
    };

    public IHttpErrorResponse Map(Error error)
        => new HttpErrorResponse
        {
            Success = false,
            ErrorCode = error.Code,
            ErrorMessage = _messages.Get(
                $"app.errors.{error.Code.ToLowerInvariant()}",
                fallback: error.Message)
        };

    public IHttpErrorResponse Map(Exception exception)
        => new HttpErrorResponse
        {
            Success = false,
            ErrorCode = "INTERNAL_ERROR",
            ErrorMessage = _messages.Get(
                "app.errors.internal",
                fallback: "An unexpected error occurred.")
        };
}

builder.Services.AddTarsLocalization();
builder.Services.AddTarsHttpErrorMapper<AppHttpErrorMapper>();
```

## Complete `Program.cs` example

```csharp
var builder = WebApplication.CreateBuilder(args);

builder.AddTarsLocalizationAspNetCore();
builder.AddTarsWebHttpOptions();
builder.AddTarsWebHttpAspNetCoreOptions();

builder.Services.AddTarsStringLocalizerSource<MyApp.Resources.SharedResource>();
builder.Services.AddTarsDefaultHttpErrorMapper();
builder.Services.AddTarsDefaultWrapDecisionService();
builder.Services.AddTarsResponseWrapperResultFilter();
builder.Services.AddTarsResponseWrapperMvcOptionsSetup();
builder.Services.AddTarsResponseWrapperEndpointFilter();
builder.Services.AddTarsExceptionFilter();

builder.Services.AddControllers(options =>
{
    options.Filters.AddService<TarsExceptionFilter>();
});

var app = builder.Build();

app.UseTarsLocalization();
app.MapControllers();

var api = app.MapGroup("/api");
api.MapGet("/orders/{id:guid}", async (Guid id, IMediator mediator, IHttpErrorMapper mapper) =>
{
    var result = await mediator.SendAsync(new GetOrderQuery(id));
    return result.ToHttpResult(mapper);
});

app.Run();
```

## Choices and trade-offs

- `ToHttpResult()` is good when each endpoint wants to explicitly decide the `IResult`.
- `ResponseWrapperResultFilter` is good when controllers should come out uniform with no repeated code.
- `ResponseWrapperEndpointFilter` is good for Minimal APIs that return simple objects, not `IResult`.
- `AddTarsProblemDetails()` complements the stack, but does not automatically replace the Tars envelope.
